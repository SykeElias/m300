Die Bedienung der Kommandozeile eines entfernten Rechners über eine Netzwerkverbindung gehört zu den grundlegendsten Elementen der Plattformadministration.

Aufruf

ssh user@host

oder

ssh -l user host

Eine Verbindung zu Host host (hostname oder ip) mit Benutzer user wird aufgebaut.

 

ssh user@host command

Eine Verbindung wird aufgebaut das Kommando command ausgeführt und die Verbindung wieder beendet.

Kopieren von Files per ssh

scp (Secure Copy)

Scp ist ein interner Mechanismus von ssh, keine weiteren Voraussetzungen sind nötig.

scp localfile user@host:remotefile (Kopie von lokal nach remote)

scp user@host:remotefile localfile (Kopie von remote nach lokal)

scp user@host1:remotefile user@host2:remotefile (Kopie von remote1 nach remote2)

Spezieller ssh Modus, der per default eingeschaltet ist. Es ist auch möglich nur sftp zu aktivieren, “normales” ssh aber nicht zu erlauben.

Zusätzliches Programm, das meist ssh als Transportweg verwendet. Es muss lokal und remote installiert sein. Es ermöglicht einfachen Abgleich ganzer Verzeichnisbäume und kann auch für Backups eingesetzt werden.

Beispiel:

Verzeichnis localdir wird mit remotedir auf host abgeglichen. Man kann den Fortschritt beobachten.

.ssh/config

Die Datei ~/.ssh/config erlaubt diverse “Shortcuts”: Damit man nicht immer alles (Hostname, User, Port, spezielles Zertifikat) angeben muss, kann man sich in dieser Datei die Shortcuts anlegen.

Nehmen wir an, auf einem RaspberryPi   (srv07.iot-lan.company.com) heisst der User Pi, und es ist ein für alle zu verwendenden Zertifikat iot-zert hinterlegt. Auch wurde Port 22 auf 2222 geändert.

sttat jedesmal ssh pi@srv07.iot-lan.company.com -i ~/.ssh/iot-zert -p 2222 anzugeben, kann die .ssh/config Datei wie folgt ergänzt werden, so dass man mit “ssh rpi-hzg” einlogen kann:

 

nano ~/.ssh/config

 

Host rpi-hzg    

  HostName srv07.iot-lan.company.com    

  User pi    

  IdentityFile ~/.ssh/iot-zert  

  Port 2222  

  IdentitiesOnly yes

    

Parameter

  • Host rpi-hzg: Definiert den Alias für die Verbindung

  • HostName: Der eigentliche Hostname/Server

  • User: Benutzername für die Verbindung

  • IdentityFile: Pfad zum privaten Schlüssel (ohne .pub!)

  • IdentitiesOnly yes: Nutzt nur die angegebene IdentityFile

  • Port: Der alternative ssh Port

Hinweise

  • Der öffentliche Schlüssel (iot-zert.pub) muss bereits im ~/.ssh/authorized_keys des RaspberryPi sein

  • Der Benutzer pi muss auf dem RaspberryPi existieren

  • Der private Schlüssel (iot-zert) sollte nicht mit .pub enden

ssh-agent / gnome-key-ring

Single Sign On SSO 

Hat man mit ssh-copyid den public Key auf einen Server kopiert, damit passwortloses einloggen funktioniert, und  eine Passphrase beim SSH-Key gesetzt, dann muss man zwar kein Passwort eintippen, aber man muss die Passphrase eingeben. 

 

 

Konzept: Der ssh-agent ist ein Hintergrunddienst, der  SSH-Passphrases zwischenspeichert (cached), sodass sie nicht bei jeder SSH-Verbindung neu eingeben werden müssen.

 

Kontrolle, ob er läuft: (Listet die Einträge auf)

ssh-add -l 

 

Starten eines Agenten, nur falls keiner läuft

eval “$(ssh-agent)”

 

Key hinzufügen:

ssh-add ~/.ssh/id_rsa # Passphrase eingeben: SML12345

 

 

Auf Ubuntu mit GUI:

  • ssh-agent wird meist von gnome-keyring gestartet
  • eval "$(ssh-agent)" wäre falsch → zweiter Agent!

SSH Tunneling

Alle Interfaces oder nichts

Bevor wir ins Tunneling einsteigen, kurz ein Review von speziellen Netzwerk Adressen:

  • 0.0.0.0
  • 127.0.0.1

Im Modul 129 (Netzwerke haben sie gelernt, dass 0.0.0.0/0 das ganze Internet bedeutet. Hier benutzen wir aber nicht ein Subnetz (kein /0), sondern wollen zum Ausdruck bringen: Alle meine lokalen Interfaces. Das heisst lo0, eth0, und weitere falls vorhanden. 

Aufgabe: Führe “ip addr” auf vmLP1 und dann auf vmLS3 aus! Wieviele Netzwerk-Interfaces sind vorhanden?

 

Beispiel 1)  Apache auf allen Interfaces

Stellen Sie sich vor, auf der Firewall vmLF1 ist ein Webserver installiert. Die vmLF1 hat drei Interfaces: WAN/RED, LAN/GREEN, DMZ/ORANGE. Auf welchen Interfaces soll der Webserver Anfragen entgegen nehmen. Vielleicht nur auf dem WAN, vielleicht aber auf allen drei. Wenn man alle drei Interfaces aktivieren möchte, schreibt man im Apache Konfig File: 0.0.0.0:80 (teilweise *:80).

 

Beispiel 2) MySql nur local

Die MySQL Datenbank soll aber aus gar keinem Netz erreichbar sein. Dann schreibt man 127.0.0.1:3306 im mysql.conf File. Man kann dann nur noch auf Mysql zugreifen, wenn man direkt auf dem Server ist und beim mysql command den Host als 127.0.0.1 oder “localhost” angibt.

 

Zusammenfassung

  • 0.0.0.0 (Zugang über alle Netzwerkkarten, eth0, wifi0, docker0, etc.)
  • 127.0.0.1 (Zugang nur vom Server aus)
  • oder die IP des Interface direkt angeben (bspw. 192.168.210.31)
Local und remote Forwarding

(siehe auch PPT)

  • Der Tunnel hat eine Richtung, einen Eingang
  • Wird der Tunnel in die “gleiche” Richtung wie der SSH Aufbau verwendet: -L (Eingang ist lokal), sonst -R (Eingang ist remote)
  • Auf beiden Seiten des Tunnels angeben, welche TCP Ports verwendet werden
  • Port-Syntax: mit -L kommt zuerst LOCAL, mit -R kommt zuerst REMOTE
    • -L locale-Interface-IP:localer-Port:remote-Interface-IP:remote-Port
    • -R remote-Interface-IP:remote-Port:locale-Interface-IP:localer-Port

Als zentraler Eintrittspunkt für Anfragen vom Internet (gemeint ist meis das Web per HTTPS) ist ein sogenanner Reverseproxy die zurzeit meistgewählte Lösung.

Funktionsweise

Ein Reverseproxy entscheidet anhand des “Host” Feldes im HTTP Request, zu welchem Webserver er die Anfrage weiterleitet.

Ab HTTP Version 1.1 muss der Client nebst dem GET <file | URL > den “Host” mitschicken. Im Beipiel unten wird “/” von www.gibb.ch verlangt:

 

Ein Reverseproxy wie Traefik entscheidet anhand dieser Host-Info, welchem Server er die Anfrage weiterschicken soll:

  • Alles mit Host= www.gibb.ch zur VM mit IP 192.168.220.11,
  • alles mit Host= www.smartlearn.ch zur VM 192.168.220.12.
Virtual-Hosts bei Apache/nginx

Ein Apache kann mit dem Host mehrere Virtual-Host betreiben, er weiss nun, ob die Anfrage von “/” oder “index.html” für die Site www.gibb.ch oder www.smartlearn.ch ist.

(Web-)Proxy

Der Vollständigkeit halber sollte auch erklärt werden, was denn der “Proxy” ist. 
Unser “Kantons-Proxy” kontrolliert den einkommenden Verkehr vom Internet. Der Proxy ist also in “unserem” Netz, wo der Firefox ist. Die Funktion des Proxy kann die Firewall übernehmen, oder eine eigene Hardware-Appliance sein. Der Browser schickt seinen Verkehr nicht zur Firewall, sondern Proxy, welcher dann auch die Daten erhält, welche vom Internet heruntergeladen werden. Der Proxy durchsucht die Daten nach Viren und Trojanern.

Zertifikate Theorie

Zertifikate erlauben 

  • die Identität eines Webservers zu überprüfen.
  • Daten verschlüsselt auszutauschen.
Verschlüsselung
  • assymetrisch (public/private key)
  • symmetrisch (beide besitzen den selben Code)
Symmetrische Verschlüsselung

Gleicher Code zum Ver- und Entschlüsseln.
1000 mal effizienter pro Byte als asymmetrische Verschlüsselung

 

Asymmetrische Verschlüsselung
2 Schlüssel => Schlüsselpaar. Das Paar besteht aus “Public Key” und “Private Key”.
Einer verschlüsselt, der andere entschlüsselt.

(ssh-keygen erzeugt ein Schlüsselpaar, id_ed25519.pub und id_ed25519)

  • Der Public Key ist im Internet öffentlich zugänglich. 
    Der Private Key bleibt lokal (=geheim).
Verschlüsseln/Entschlüsseln 

Man kann zuerst mit dem Public Key oder dem Private Key verschlüsseln:

  • Mit Public Key verschlüsselt →Nur mit Private Key zu entschlüsseln.
  • Mit Private Key verschlüsselt →Nur mit Public Key zu entschlüsseln.
Idee Koffer mit Spezialschloss

Das Schloss kennt drei Stellungen: Links, Mitte, Rechts. Links und Rechts ist verschlossen, in der Mitte offen.

Der rote Schlüssel lässt sich nur nach links drehen, der blaue nur nach rechts. Wird mit dem Roten die Truhe verschlossen, kann die Truhe nur mit dem Blauen wieder geöffnet werden. Wird aber mit dem Roten verschlossen, kann nur mit dem Blauen geöffnet werden.

Verschlüsseln A sendet zu B

A verschlüsselt die Daten mit dem public K von B → B entschlüsselt mit seinem private K. B weiss “Das kann nur für mich sein, schliesslich habe nur ich den privaten Key”. (Ich weiss aber nicht wirklich, wer es geschickt hat, schliesslich kennt jeder meinen Public Key).

Signieren

A verschlüsselt mit seinem Private Key → B (oder jeder andere) entschlüsselt mit dem Public von A. Der Empfänger weiss “Das kann nur von A kommen, nur er hat den passenden privaten Key zum public-Key von A”. 

 

Verschlüsseln und signieren

Damit A weiss, dass nur B es entschlüsseln kann, und B sicher ist, dass die Daten von A stammen, wird zuerst verschlüsselt, und dann das Packet signiert. 

Computational expensive

Asymmetrische Verschlüsselung ist rechenintensiv, ca. 1000 mal langsamer als symmetrische.

Daher werden die Daten nicht assymmetrisch verschlüsselt, sondern es wird ein Code (=Key) für ein symmetrisches Verfahren (bspw. AES256) zufällig erzeugt, nur dieser Code wird assymmetrisch verschlüsselt und ausgetauscht. Die folgenden Daten werden dann mit dem symmetrischen Verfahren ver- und entschlüsselt. 

Bei der Signatur wird zuerst ein Hash des Dokument gerechnet (z.Bsp. sha256) und dieser Hash (=Fingerprint) wird danach signiert und dem Dokument angehängt.

Zertifikatsverifikation bei HTTPS / TLS

Der Browser (Firefox) wird einen zufälligen Code erzeugen, um mit einem symmetrischen Verfahren die Daten mit dem Webserver gesichert auszutauschen.

Der Browser benötigt dazu den Public Key vom Server.

Daher schickt der Webserver ganz zu Beginn dem Client sein Zertifikat, welches aus Meta-Daten plus seinem Public-Key besteht.

Nun könnte aber ein Man-in-the-middle den Public-Key mit seinem ersetzt haben. Der Browser muss also überprüfen können, dass der Public-Key wirklich dem Webserver gehört.

 

Intermediate-CA und Root-CA

Eine “Certificate Authority” (CA) ist eine “Behörde”, welche Zertifikate bestätigt. Sein Key-Paar für das Zertifikat hat der Webserver selber erzeugt, er will ja den Private-Key niemandem preisgeben. Hingegen will er aber, dass eine CA sein Zertifikat signiert, das hesst, bestätigt, dass er effektiv ist, wen er vorzugeben scheint. 

Certificate Signing Request CSR

Die CA erhält also vom Webserver einen Antrag, sein Zertifikat zu signieren. Die CA rechnet den Hash des Zertifikat und verschlüsselt den Hash mit ihrem Private-Key. Damit der Browser diesen entschlüsseln kann, benötigt er also den Public-Key der CA. Den Public-Key schickt die CA auch grad mit. Nun hat der Browser aber wieder das selbe Problem: Wieso der CA trauen? 

Root-CA Public Certs im Browser

Die Lösung ist, dass der Webserver den Public-Key der CA auch als signiertes Zertifikat mitschickt, diesmal ist das CA-Zertifikat aber von einer Root-CA und dem Private-Key der Root-CA unterschrieben. Und die Public-Keys der Root-CAs sind in den Browsern enthalten.

Der Browser kann also mithilfe seiner Liste von Root-CAs Public-Keys das Zertifikat der Intermediate-CA validieren, und hat somit sicher den richtigen Public-Key der Intermediate-CA. Mit dem Public-Key der Intermediate-CA kann nun das Zertifikat und der Public-Key des Webservers validiert werden.

Let's encrypt

Nachdem Edward Snowden als NSA Mitarbeiter 2013 geheime Dokumente veröffentlichte, die zeigten, dass der US-Geheimdienst versuchte den gesamten Internetverkehr mitzuhören, wurde die Verschlüsselung mit HTTPS zum Standard. 

 

Lets Encrypt ist ein Dienst/Organisation, welche kostenlose Zertifikate ausstellt https://letsencrypt.org/ und seit 2014 aktiv ist. Fast 80% des Web-Datenverkehrs ist mit Zertifikaten von Lets Encrypt verschlüsselt.

ACME

ACME (Automatic Certificate Management Environment)

Die Überprüfung und Ausstellung wird automatisiert ausgeführt, da ein Zertifikat nach Ablauf der Gültigkeitsdauer erneut ausgestellt und installiert werden muss. (Ein Let's Encrypt Zertifikat hat im Vergleich zu anderen Ausstellern nur eine kurze “Lebensdauer” von ein paar Wochen statt ein bis drei Jahre).

ACME API
Der Antrag, die Überprüfung und die Ausstellung geschehen über eine genau definierte API (https://datatracker.ietf.org/doc/html/draft-ietf-acme-acme-09)

Diverse Programme implementieren diese API. Für uns relevant ist der Certbot und LEGO (bei Traefik eingesetzt), evt. acme.sh (Eine gute Zusammenstellung von Software-Implementationen:https://acmeclients.com/)
 

Authentifizierung / Challenges

Wichtig ist zu wissen, wie Lets Encrypt überprüft, ob man für den (oder die) gewünschten Domains/Hostnamen authorisiert ist, ob man diese wirklich besitzt.
Dies geschieht mit sogenannten Challenges: Ein von Lets Encrypt vorgegebener Token muss auf der eigenen Domain installiert werden, um zu beweisen, dass man sie “besitzt”. Dies wird vor der Zertifikatsausstellung geprüft.

Es gibt verschiedene Typen von Challenges:
https://letsencrypt.org/docs/challenge-types/

 

Wir beschränken uns auf häufigst genutzten Typen:

  • HTTP-01 challenge
  • DNS-01 challenge
  • TLS-ALPN-01

(TLS-ALPN-01 ist für die Verwendung bei Traeffik die beste Wahl)

 

  • Bei der HTTP-01 Challenge muss man unter http://meinedomain/.well-known/acme-challange/ ein File anlegen, und dieses muss von dem von Letsencrypt vorgegeben Inhalt sein. Let'sencrypt  macht ein wget http://meinedomain/.well-known/acme-challange/<XYZ> und überprüft, ob der Inhalt des Files korrekt ist.
  • Bei der DNS-01 Challange muss ein TXT Record auf dem DNS Server (in der Zone der Domain) hinzugefügt werden. Die Überprüfung erfolgt durch “dig TXT _acme-challenge.<meinedomain>” . Nach Ausstellen des Zertifikats wird der TXT Record wieder entfernt.
